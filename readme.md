# 🛒 AI-Powered Conversational E-Commerce Search (v0.2)

> 🚧 **Note:** This project is currently in active development (Work in Progress). Architecture decisions, models, and system flows are subject to change as the prototype evolves.

## 🎯 Project Overview
This project aims to replace traditional keyword-based (BM25) e-commerce search engines with an AI-powered product discovery engine that understands user intent and context. Instead of relying on exact matches, the system processes natural language queries (e.g., *"My dad is a former sailor, I need a meaningful birthday gift recommendation"*) to expand the semantic search space and retrieve products with the highest contextual relevance.

## 🏗️ System Architecture & Search Flow

The system utilizes a dual-model LLM pipeline and a two-stage filtering/ranking mechanism to maximize search accuracy and performance:
### 0. Product Ingestion & Auto-Query Generation (Offline Phase)
Before a user even performs a search, the system prepares the catalog during the product upload phase. When a store administrator adds a new product:
* **Synthetic Question Generation:** The LLM analyzes the product's details (title, description, category) and automatically generates **3 potential natural language questions** a user might ask to find this exact item (e.g., generating *"What is a good nostalgic gift for a retired sailor?"* for a vintage compass).
* **Vectorization:** These 3 synthetic questions, along with the product data, are converted into high-dimensional embeddings.
* **Storage:** The generated embeddings are stored directly into the **pgvector** database.
* **Why this matters:** This process drastically expands the "semantic surface area" of every product. During a live search, the user's query is matched against these pre-calculated synthetic questions rather than just dry product descriptions, ensuring a much higher semantic match rate.

### 1. Query Enrichment (Phi-3)
The user's natural language query is first processed by the **Phi-3** model. The model's objective is to understand the user's intent, extract emotional context, and generate alternative **semantic search queries** and **product concepts** based on a single input sentence.

### 2. Structured Data Parsing (Flan-T5 Base)
The flexible, text-based output generated by Phi-3 is then passed to the **Flan-T5 Base** model. The sole purpose of this secondary model is to format the data into a strict, predefined JSON schema that the backend can parse without errors.

*Standard JSON Output generated by the system:*
```json
{
  "is_query_valid": true,
  "clarification_message": null,
  "metadata_filter": {
    "special_event": "fathers_day",
    "style": "nostalgic",
    "gender": "male",
    "in_stock": true,
    "price_min": null,
    "price_max": null
  },
  "inferred_product_concepts": [
    "nautical themed vintage compass",
    "personalized captain plaque",
    "collectible ship model"
  ],
  "expanded_semantic_queries": [
    "gift ideas for former sailor dad",
    "nostalgic nautical themed decorative items",
    "meaningful birthday gift for men"
  ],
  "product_synthetic_queries": [
    "For which users would this product be a meaningful gift?",
    "Why would nautical collection enthusiasts prefer this?",
    "How is this positioned in the nostalgic decoration category?"
  ]
}
```

### 3. Pre-Filtering (SQL Metadata Filtering)
To reduce the computational cost and error rate of the vector search, an initial elimination is performed at the SQL level. Using the exact data in the `metadata_filter` object (e.g., `gender: male`, `in_stock: true`), the target product pool is narrowed down. Irrelevant products are filtered out before entering the vector space.

### 4. Vector Comparison (Semantic Matching)
Within the narrowed product pool, the user's dynamically `expanded_semantic_queries` are compared against pre-generated synthetic product queries (`product_synthetic_queries`) using **PostgreSQL (pgvector)**. 

### 5. Scoring & Ranking
Products are sorted based on the **similarity scores (match percentages)** obtained from the vector comparison. The items that capture the highest semantic alignment with the user's expanded intent are presented at the top of the search results.

## 💻 Tech Stack
* **Backend:** ASP.NET Core
* **Database:** PostgreSQL + `pgvector`
* **AI / LLM Pipeline:** * Query Enrichment: **Phi-3**
  * JSON Validation: **Flan-T5 Base**
* **Frontend:** Angular *(Integration in progress)*

## 🚀 Getting Started (Development Environment)
1. Clone the repository.
2. Install PostgreSQL and enable the `pgvector` extension.
3. Configure your database connection strings in the `.NET` project's `appsettings.json` file.
4. Run the API using the `dotnet run` command.
